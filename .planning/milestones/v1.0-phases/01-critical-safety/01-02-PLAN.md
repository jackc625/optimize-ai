Create and apply a SQL migration that enables Row Level Security and creates four access policies on all nine Supabase tables. This closes the data isolation gap where one authenticated user could potentially read or modify another user's rows.

Purpose: RLS is the database-level guarantee that every user sees only their own data. Middleware and application-level checks are defense-in-depth, but RLS is the authoritative security boundary.
Output: `supabase/migrations/20260226_rls_all_tables.sql` — idempotent SQL covering all 9 tables. Applied to the Supabase database.

SQL pattern per table (from RESEARCH.md):
```sql
-- Idempotent: drop before create
DROP POLICY IF EXISTS "table_name: users can select own rows" ON public.table_name;
DROP POLICY IF EXISTS "table_name: users can insert own rows" ON public.table_name;
DROP POLICY IF EXISTS "table_name: users can update own rows" ON public.table_name;
DROP POLICY IF EXISTS "table_name: users can delete own rows" ON public.table_name;

ALTER TABLE public.table_name ENABLE ROW LEVEL SECURITY;

CREATE POLICY "table_name: users can select own rows"
ON public.table_name FOR SELECT TO authenticated
USING ((SELECT auth.uid()) = user_id);

CREATE POLICY "table_name: users can insert own rows"
ON public.table_name FOR INSERT TO authenticated
WITH CHECK ((SELECT auth.uid()) = user_id);

CREATE POLICY "table_name: users can update own rows"
ON public.table_name FOR UPDATE TO authenticated
USING ((SELECT auth.uid()) = user_id)
WITH CHECK ((SELECT auth.uid()) = user_id);

CREATE POLICY "table_name: users can delete own rows"
ON public.table_name FOR DELETE TO authenticated
USING ((SELECT auth.uid()) = user_id);
```

Note: `(SELECT auth.uid())` (subquery form) is the Supabase-recommended pattern — evaluates once per query, not once per row. Do NOT use bare `auth.uid()` without the SELECT wrapper.
