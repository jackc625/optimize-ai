# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Type Safety.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Important context:** Phase 2 is a backend/refactoring hardening pass.
> No new screens, layouts, or visual components are introduced.
> This contract defines the existing visual language that MUST be preserved
> during refactoring. Every component touched in this phase must exit with
> identical visual output to its pre-refactor state.

## Spacing Scale

Declared values (must be multiples of 4). Source: Card.tsx and form components
establish these as the in-use tokens.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, tight inline padding |
| sm | 8px | Compact padding — e.g. `py-1`, `px-2` inputs in MacroSummary |
| md | 16px | Default element spacing — e.g. form `space-y-4` gap |
| lg | 24px | Section padding — e.g. `px-6 py-6` in CardContent |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions: `px-3 py-2` (12px / 8px) used for form inputs — preserved as-is,
do not normalize during refactor.

## Typography

Derived from `globals.css` (`font-sans` = Inter), `Card.tsx` (`text-xl font-semibold`),
and `MacroSummary.tsx` (`text-sm`).

| Role | Size | Weight | Line Height | CSS Class |
|------|------|--------|-------------|-----------|
| Body | 16px (text-base) | 400 (regular) | 1.5 | `text-base` |
| Label | 14px (text-sm) | 400 (regular) | 1.5 | `text-sm` |
| Card heading | 20px (text-xl) | 600 (semibold) | 1.2 | `text-xl font-semibold` |
| Display | 28px (text-2xl) | 600 (semibold) | 1.2 | `text-2xl font-semibold` |

Four sizes declared. Two weights: 400 (regular) and 600 (semibold).

Note: `font-medium` (weight 500) currently appears on label elements in
`ProfileForm.tsx`, `MacroSummary.tsx`, `Button.tsx`, and page files. These
instances are outside the two-weight contract. Phase 2 executors must normalize
all `font-medium` label instances to `text-sm` (weight 400) when touching those
files as part of type-safety refactoring. `Button.tsx` uses `font-medium` for
button text — that instance should be updated to `font-semibold` to align with
the 600 weight on interactive primary elements.

## Color

All colors are CSS custom properties resolved at runtime. Values below are the
light-mode `:root` defaults from `globals.css`.

| Role | CSS Variable | Light Value | Usage |
|------|-------------|-------------|-------|
| Dominant (60%) | `--background` | `hsl(0 0% 100%)` — white | Page background, card surfaces |
| Secondary (30%) | `--card` / `--muted` | `hsl(0 0% 100%)` / `hsl(0 0% 96.1%)` | Cards, input backgrounds, muted surfaces |
| Accent (10%) | `--primary` | `hsl(0 0% 9%)` — near-black | Primary buttons, focus rings — see reserved list |
| Destructive | `--destructive` | `hsl(0 84.2% 60.2%)` — red | Destructive actions only |

Accent (`--primary`) reserved for:
- `Button` variant="primary" background
- Form input focus ring (`focus:ring-primary-300`)
- "Recalculate" and "Save" button backgrounds in MacroSummary
- "Save Changes" / "Set Up Profile" submit button in ProfileForm

Accent is NOT used for body text, labels, card borders, or neutral interactive states.

`--muted-foreground` (`hsl(0 0% 45.1%)`) is used for secondary/helper text in
loading and empty states — this is a separate semantic role, not accent.

## Interaction Contracts

### useMacros "Recalculate" button

**Before migration:** `await refresh()` — calls `useState`/`useEffect` setter.
**After migration:** `await refetch()` from `useQuery` return value.
**Visual contract:** Button appearance, disabled state, and "Recalculating…"
copy must be identical. No visual change is permitted.

### Form submit (ProfileForm)

**Before:** `ProfileSchema.parse(parsedInput)` after `as` casts.
**After:** `ProfileSchema.parse(parsedInput)` on raw strings (casts removed).
**Visual contract:** Form renders identically. Validation errors continue to
surface as toast messages using `err.errors[0].message`.

### Habit page consumption

**Before:** `const { habits, todayCompleted, loading, addHabit } = useHabits()`
**After:** `const { data, isLoading, error } = useHabits()` + `const { mutate: addHabit } = useAddHabit()`
**Visual contract:** Habit list renders identically. Loading state maps `isLoading`
to the same loading UI. `todayCompleted` accessed as `data?.todayCompleted ?? []`
(string array) then converted to `new Set()` in consuming component if O(1) lookup
needed.
