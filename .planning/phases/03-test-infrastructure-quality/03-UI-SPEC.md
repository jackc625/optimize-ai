## Spacing Scale

Standard 8-point scale. All values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline button padding |
| sm | 8px | Gap between buttons in a group (e.g. CardFooter button row) |
| md | 16px | Default element spacing, card content padding supplement |
| lg | 24px | Card padding (`px-6` = 24px, `py-4` = 16px — matches existing Card.tsx) |
| xl | 32px | Layout section gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical spacing |

Exceptions: Touch targets for interactive icon-only elements (delete button on habit rows) must be minimum 44px — use `min-h-[44px] min-w-[44px]` or ensure the surrounding `<li>` row provides adequate tap area.

Source: Derived from existing Card.tsx (`px-6 py-4`, `px-6 py-6`). Default: 8-point scale standard.

## Color

All color values reference CSS custom properties from `globals.css`. Do not use hex values directly in component code.

| Role | CSS Variable | Light Value | Usage |
|------|-------------|-------------|-------|
| Dominant (60%) | `--background` / `bg-background` | `hsl(0 0% 100%)` — white | Page backgrounds, full-screen containers |
| Secondary (30%) | `--card` / `bg-card`, `--muted` / `bg-muted` | `hsl(0 0% 100%)` / `hsl(0 0% 96.1%)` | Card surfaces, input backgrounds, skeleton fill, sidebar |
| Accent (10%) | `--primary` / `bg-primary` | `hsl(0 0% 9%)` — near-black | Primary CTA buttons only |
| Destructive | `--destructive` / `bg-destructive` | `hsl(0 84.2% 60.2%)` — red | Destructive confirm button in ConfirmDialog, delete action text |

Accent (`--primary`) reserved for: **Primary CTA buttons only** (e.g. "Add Habit", "Log Weight", "Save", "+ New Workout"). Not used for links, headings, or neutral interactive elements.

Destructive (`--destructive`) reserved for: **ConfirmDialog confirm button (destructive variant) only**. Delete action text uses `text-destructive` (foreground semantic token). Do not apply destructive color to warning states or informational error messages.

Source: `globals.css` CSS variable declarations. Usage pattern derived from existing page components.

## Component Inventory

New components introduced in this phase. Each must adhere to this specification.

### `<Skeleton />` — `src/components/ui/Skeleton.tsx`

- Renders a single `<div>` with `animate-pulse rounded bg-muted`
- Accepts `className` prop for sizing — caller controls `h-*` and `w-*`
- No children — purely presentational
- `cn` import path: `@/lib/utils` (confirmed — `src/lib/utils.ts` exports `cn`)
- Reference pattern: habits page inline skeleton (lines 63–72 of `habits/page.tsx`)

**Skeleton shapes per page:**

| Page | Loading state to replace | Skeleton shape |
|------|--------------------------|---------------|
| Workouts list | `<div className="text-sm text-muted-foreground">Loading workouts…</div>` | 3 skeleton cards in `grid grid-cols-1 md:grid-cols-2 gap-4`. Each card: header `h-6 w-48`, body `h-4 w-32 mt-2`, footer row 3x `h-8 w-16` |
| Weight log | `<p className="text-muted-foreground text-center">Loading…</p>` inside "Weight Logs" Card | 3 skeleton rows: `flex justify-between py-2 border-b border-border`, each side `h-4 w-24` |
| Habits | Inline `animate-pulse` list (lines 63–72) | Refactor to use shared `<Skeleton />`: 3 `<li>` items with `flex items-center space-x-4`, containing `h-4 w-32`, `h-4 w-12`, `h-5 w-5 rounded-full` |

Item count: exactly 3 skeleton items per list (matches CONTEXT.md decision).

### `<ConfirmDialog />` — `src/components/ui/ConfirmDialog.tsx`

Props interface (locked):
```typescript
interface ConfirmDialogProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  title: string;
  description: string;
  confirmLabel?: string;     // default: "Confirm"
  onConfirm: () => void;
  variant?: 'destructive' | 'default';
}
```

Visual specification:
- **Overlay:** `fixed inset-0 bg-black/50 z-40` — semi-transparent dark backdrop
- **Content panel:** `fixed left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2 z-50 bg-card rounded-[var(--radius)] border border-border shadow-lg p-6 w-full max-w-md`
- **Title:** `text-xl font-semibold text-foreground` (matches CardTitle pattern)
- **Description:** `text-sm text-muted-foreground mt-2`
- **Button row:** `flex gap-3 justify-end mt-6`
- **Cancel button:** `<Button variant="outline">Cancel</Button>` — neutral gray
- **Confirm button (destructive):** `<Button>` with className override `bg-destructive text-destructive-foreground hover:bg-destructive/90` — red
- **Confirm button (default):** `<Button variant="primary">` — near-black

Button.tsx note: The `destructive` variant does not exist in `Button.tsx` (confirmed — only `primary`, `outline`, `ghost`). Use `className` override on the confirm button for destructive variant rather than adding a new variant to Button.tsx (out of phase scope).

Interaction behavior:
- Esc key closes dialog (Radix AlertDialog handles this natively)
- Focus trapped inside dialog while open (Radix AlertDialog handles this natively)
- Clicking overlay does NOT close — Radix AlertDialog.Root behavior; cancel must use the Cancel button
- Tab cycle: Cancel → Confirm → Cancel (logical order for destructive action: cancel first)

### `logError()` — `src/utils/logger.ts`

Not a UI component — no visual contract needed. Logging only affects browser DevTools console output.

## Interaction States

### Skeleton loading

- Skeleton renders immediately when `isLoading` is `true`
- No spinner or "Loading…" text alongside skeleton (replace, not supplement)
- Skeleton count is fixed at 3 regardless of eventual data count
- Transition from skeleton to loaded state: no animation required — DOM replace is sufficient

### ConfirmDialog

- Trigger: user clicks "Delete" button on a workout card
- Dialog opens immediately (no delay)
- While dialog is open: page content behind overlay is still visible but dimmed
- After "Delete Workout" confirm: dialog closes, `deleteWorkout.mutate(id)` fires, existing toast pattern handles success/error feedback
- After "Cancel": dialog closes, no mutation fires
- Delete button in workout card shows "Deleting…" text when `deleteWorkout.isPending` is true (existing behavior — retain)

### Error logging (logError)

- No visible UI change — logError is a developer-facing utility only
- `toast.error()` continues to be the user-facing error notification (existing pattern — retain)
- logError replaces the `console.error()` call at the same call site
